Section 02
产品动态
AI 消费级产品
Claude Anthropic · Opus 4.8 Effort Control 2026.05.28
  • 可控努力等级（Effort Control）：claude.ai 新增「努力旋钮」，用户可选择高 / 更高 / 最高三档，默认高档；Claude Code 支持 xhigh / max 参数——AI 的工作量第一次成为用户可设置的显式参数。
  • Claude Code 动态工作流（Dynamic Workflows）：应对超大规模问题时，Opus 4.8 可自主分解任务、协调子工作流，不再受限于单轮上下文——从「一问一答」到「项目级自主执行」。
  • 快速模式（Fast Mode）：以 2.5× 速度运行，Pro 用户免费试用；开发者可通过 API 以 $10 / $50 per million tokens 使用——响应速度与质量之间有了可见的权衡旋钮。
「努力等级」是 AI 交互里一个全新的设计语言：用户不再只能接受 AI 的默认输出质量，而是能显式地表达「我想要更多思考」还是「我想要更快答案」。这为未来产品引入「AI 深度 vs. 速度」的显式切换提供了范例——在工具型产品里，这种控制感本身就是信任建立的手段。
来源：anthropic.com/news/claude-opus-4-8
ChatGPT OpenAI · 职业平台 + GPT-5.5 Instant 升级 2026.05.29–06.02
  • ChatGPT 职业平台（Jun 2）：整合 Indeed / Upwork / Appcast 职位源，基于用户经验和技能个性化推荐职位；配套简历创建、上传、AI 针对职位定制优化、一键专业格式下载——美国用户所有套餐可用。
  • GPT-5.5 Instant 可读性升级（May 29）：响应更自然流畅，减少过度使用列表的习惯，去除 Canvas 功能；o3 将于 8 月 26 日退出 ChatGPT，GPT-4.5 于 6 月 27 日退役。
  • OpenAI Codex 扩展角色插件（Jun 2）：Codex 接入角色专属插件，正向知识工作者的通用 App 方向演进——不只是编程助手，而是面向产品 / 设计 / 运营等角色的 Agent 平台。
ChatGPT 切入求职场景是一个典型的「平台化扩张」信号：AI 助手不再满足于回答问题，而是开始接管用户生命周期中的高频、高价值场景（找工作、写简历）。对产品设计的启示：当 AI 整合了外部数据源（职位库），原本需要跨应用操作的流程开始在单一入口完成，这对传统垂类工具的流量入口形成直接竞争。
来源：the-decoder.com · ChatGPT Career · GPT-5.5 Instant 升级
Gemini Google · Gemini Omni + Gemini 3.5 Flash 2026.05.28–29
  • Gemini Omni——任意输入生成视频：Google I/O 2026 后续落地，Omni 可同时接受图片 / 音频 / 视频 / 文字作为输入，生成扎根真实世界知识的高质量视频；支持对话式剪辑，不需要传统时间轴操作。
  • Gemini Omni Flash 向订阅用户推送：向 AI Plus / Pro / Ultra 全球用户开放，YouTube Shorts 和 YouTube Create App 免费可用——视频创作能力首次进入大众消费场景。
  • Gemini 3.5 Flash——Agent 工作流的前沿小模型：在 coding 和复杂 agentic 任务上取得前沿水平，专为「长时运行、多工具调用」场景优化；前端已有 9 个演示视频展示 Omni + 3.5 Flash 协作能力。
  • Google I/O 2026 十二大亮点回顾（May 28）：Gemini Spark 日简报、Universal Cart 全站购物车、Daily Brief、智能眼镜、SynthID 水印……消费级 AI 的「超级应用」形态轮廓已清晰。
Gemini Omni 打破了视频创作工具的「输入单一性」假设——原来视频生成需要专属的文字/图片入口，现在任何已有素材（一段录音、一张截图、一个视频片段）都可以成为起点。对多媒体内容产品的 UX 启示：上传入口的设计应当拥抱「任意素材」，而不是为不同格式设置独立入口。
来源：blog.google · Gemini Omni & 3.5 demos · I/O 2026 十二大亮点
Section 02 · B
企业级 AI 产品 & 云平台
企业级 AI · Build 2026
Microsoft 365 Work IQ · Frontier Tuning · Build 2026 2026.06.02
  • Work IQ——生产级 Agent 智能层：10 个通用 MCP 工具打通 M365 全数据（邮件 / 日历 / 文件 / 人员 / 聊天 / 站点），Agent 通过标准动词（fetch / create / update）操作数据，结合 getSchema 动态发现数据结构——6 月 16 日 GA。
  • Frontier Tuning——企业数据驱动的 Agent 调优：用企业自有数据（会话记录 / 知识库 / M365 文档）持续调优 Agent，在 Copilot Studio 和 Microsoft Foundry 内直接接入；已与 EY / Pearson / McKinsey 等大型企业完成验证。
  • Build 2026 协作 Agent 框架：「Build collaborative agents where work happens」——Agent 在 Teams / Outlook / SharePoint 这些工作实际发生的地方协作，而不是在独立 AI 界面里孤立运行。
Work IQ 的核心设计决策是「10 个工具打通一切」——不是为每个 M365 应用单独构建 API，而是用极小的接口面通过 MCP 协议统一抽象。对企业产品设计的启示：当 Agent 成为数据消费者，接口设计的「最小集」原则比「功能齐全」更重要——冗余接口会增加 Agent 的决策负担，导致行为不可预测。
来源：devblogs.microsoft.com · Work IQ · Frontier Tuning
GitHub Copilot Agent-Native Desktop App · Canvas · Build 2026 2026.06.02
  • Copilot 全新桌面应用——Agent 原生开发控制中心：不再是 IDE 插件，而是独立桌面应用；核心理念是「让 Agent 在你已有的工作流中运行」——文件系统、终端、浏览器、API 都在一个统一界面协同。
  • Canvas——意图变为可检视的工作输出：Agent 执行过程在 Canvas 中实时展现，每个步骤的操作、文件变更、工具调用均可见；用户不再面对「黑盒 Agent」，而是可以在任意检查点接管或调整方向。
  • 适配 agentic 输出的代码审查：Agent 大批量生成代码时，传统逐行 Code Review 难以扩展；新 Code Review 以「意图 + 变更摘要」为主视图，人工只需审核高风险或不确定的部分——审查粒度从代码行升级到决策点。
  • 统一 apps / tools / agents 的单一运行时：Copilot App、CLI 工具、自定义 Agent 共享同一运行时，开发者不再为不同 Agent 形态维护多套基础设施。
GitHub Copilot Canvas 是本期「Agent 工作可见性」设计中最完整的实现案例。它回答了一个核心 UX 问题：当 Agent 替你工作时，你怎么知道它在做对的事？Canvas 的答案是「全程可见 + 随时可接管」——不是事后的日志，而是实时的共享视图。对我们产品里 Agent 功能的设计启示：可见性（Visibility）和可接管性（Takeover）需要作为一等需求进行设计，而不是事后追加的日志窗口。
来源：github.blog · GitHub Copilot App
Section 03
设计工具动态
设计工具 & 创作平台
Figma Make · 本地代码集成 2026.05.28
  • Figma Make 连接本地代码库：Make 的 AI 能力从 Figma 云端延伸到开发者的本地代码仓库，视觉编辑、上下文 prompting 和协作均可直接作用于本地代码——设计侧改动即时同步至开发环境。
  • 双向协同：设计与代码真正对齐：开发者可在 Figma Make 里口述修改意图，AI 识别上下文后在本地代码中应用变更；设计师看到的改动和工程师看到的代码变更来自同一次操作。
  • 系列配套：Config 2026 收官之作：Make 本地代码是 Figma Config 2026（5 月 20 日 AI Design Agent 发布）的开发者配套——Agent 处理设计层，Make 负责代码层，两层协同完成设计到实现的完整链路。
Make 本地代码打通了设计到开发之间最难跨越的那道鸿沟——不是「将设计稿导出给开发」，而是「设计师和开发者在同一 AI 辅助环境里共同操作同一份实现」。对设计系统团队的启示：Design Token 的价值在这个场景里被极大放大——只有有良好结构化 Token 定义的设计系统，才能在 AI 驱动的跨层协同中保持一致性。
来源：figma.com/blog · Make 本地代码
Mistral Vibe 远程 Agent + Le Chat Work Mode + Mistral Medium 3.5 2026.05.28
  • Vibe 远程 Agent——Coding Agent 迁移至云端：原本在本地运行的 Vibe Coding Agent 现在在云端异步并行运行，用户从 CLI 或 Le Chat 对话中启动后可以离开——Agent 自行完成任务，完成后通知用户，打破「等待 AI 完成」的交互模式。
  • Le Chat Work Mode：基于 Mistral Medium 3.5 的多步骤任务 Agent，接入工具并行调用，适合研究、分析、长周期工作任务；Le Chat 从单轮对话工具升级为可异步执行复杂任务的「工作伙伴」。
  • Mistral Medium 3.5 开源发布：128B 稠密模型，指令跟随 / 推理 / 编程单一权重合一，SWE-Bench 评分 77.6%，仅需 4 块 GPU 可自托管；开放权重使企业可在本地私有化部署同等能力。
Vibe 远程 Agent 代表了一个重要的 UX 范式转变：AI 任务的执行从「同步阻塞」（用户等待）变成「异步通知」（用户去做别的，AI 完成后告知）。这在移动端尤其有意义——手机上用户不会盯着屏幕等几分钟，异步推送通知才是正确的交互形态。对我们产品里的长时 Agent 任务，异步化 + 进度通知应当成为标准 UX 模式，而不是让用户在 loading 状态里干等。
来源：mistral.ai/news · Vibe Remote Agents & Medium 3.5
Section 04
对我们的启示
启示 01
Claude Opus 4.8 · 努力等级控制
「AI 努力程度」成为可设计的用户参数
Claude 的努力等级旋钮是 AI 交互里一个新的透明度设计范式：用户不再被动接受 AI 的默认质量，而是显式表达「我要更深的思考」或「我要更快的结果」。这种控制感本身就是信任建立的手段。对我们产品的启示：凡是有「AI 深度 vs. 速度」权衡的功能，都值得考虑将这个选择暴露给用户——哪怕只是简单的「精准模式 / 快速模式」切换，也能显著提升用户对 AI 输出的主人感。
启示 02
ChatGPT 职业平台 · AI 场景泛化
AI 助手正在平台化——每个垂类场景都是潜在入口
ChatGPT 接入职位库做求职 + 简历，是 AI 助手向「超级应用」演进的又一步：不是构建专属求职产品，而是将外部数据源（Indeed / Upwork）接入对话界面，让原本需要跨应用的流程在单一入口完成。对我们的启示：当产品与 AI 深度整合时，原本「用户需要去别处完成」的流程（查数据、填表单、找文档）都是可以被 AI 接管、在产品内完成闭环的机会。评估自己产品里有多少「用户需要离开去做的事」，就有多少 AI 整合的空间。
启示 03
Gemini Omni · 多模态视频创作
任意素材皆可成为创作起点——输入入口需要重新设计
Gemini Omni 打破了视频创作工具「从零开始输入文字」的假设——任何已有素材（录音片段、截图、视频剪辑）都可以成为生成起点，且支持对话式剪辑，无需传统时间轴。这对内容创作产品的 UX 提出了新要求：上传入口的设计应当拥抱「任意格式」，而不是为每种输入格式设置独立、割裂的入口。在我们产品中，凡是有 AI 生成功能的地方，都应当思考：用户可以从哪些已有的材料「续写」，而不只是「从空白开始」。
启示 04
Microsoft Work IQ · MCP 标准协议
Agent 接口设计：「最小集」优于「功能齐全」
Work IQ 的核心决策是用 10 个通用 MCP 工具打通整个 M365 数据层——不是为每个应用单独构建 API，而是用极小的接口面统一抽象。这是 Agent 时代接口设计的重要范式：冗余工具会增加 Agent 的决策负担，导致行为不可预测、出错率升高。对产品设计的启示：为 AI Agent 设计工具接口时，宁可极度克制，只暴露最核心的能力，不要把「功能完整」当成目标。Agent 能用少量工具完成复杂任务，比拥有大量专用工具但行为混乱要好得多。
启示 05
GitHub Copilot Canvas · Agent 工作可见性
可见性与可接管性——Agent 功能的两个设计必答题
GitHub Copilot Canvas 回答了一个核心 UX 问题：当 Agent 替你工作时，你怎么知道它在做对的事？答案是「全程可见 + 随时可接管」——不是事后日志，而是实时共享视图，用户可在任意检查点接管或调整方向。对我们产品里所有 Agent 功能的设计要求：可见性（Visibility，用户能看到 Agent 在做什么）和可接管性（Takeover，用户随时能暂停 / 修正 / 接手）需要作为一等需求来设计，而不是在功能上线后追加的「日志面板」。没有这两点，用户对 Agent 的信任无从建立。
启示 06
Mistral Vibe 远程 Agent · 异步执行模式
长时 Agent 任务应当「异步化 + 通知化」，不应让用户干等
Mistral Vibe 将 Coding Agent 迁移到云端异步运行，用户启动后可以去做别的事，Agent 完成后推送通知——这是对「用户等待 AI 完成」交互模式的根本性改变。在移动端这一点尤为关键：用户不会盯着手机屏幕等几分钟。对我们产品里所有执行时间超过 30 秒的 AI 任务，异步化 + 进度通知应当成为标准设计模式。这不只是技术架构问题，而是用户体验选择：让用户等待 loading 是最低效的时间使用方式，也是信任损耗最快的交互形态。
